# HTTP 메서드

HTTP API 만들기.

### 요구사항
* 회원 목록 조회
* 회원 조회
* 회원 등록
* 회원 수정
* 회원 삭제

### API URI 설계
일단 설계를 해보자.
* 회원 목록 조회 
  * /read-member-list
* 회원 조회 
  * /read-member-by-id
* 회원 등록 
  * /create-member
* 회원 수정 
  * /update-member
* 회원 삭제 
  * /delete-member

이게 좋은 방법일까?

HTTP API 설계 시 가장 중요한 것은 \
**리소스**\
이다.

### 리소스의 의미
* 리소스는 회원이라는 개념 자체가 바로 리소스.
* 리소스는 어떻게 식별해야 하는가.
  * 회원 등록/수정/조회는 모두 잊는다.
  * 회원이라는 자체만 식별하면 됨.
  * 그렇기에 회원 리소스를 URI에 매핑

리소스를 식별한 설계는 아래와 같다.
* 회원 목록 조회
    * /members
* 회원 조회
    * /members/{id}
* 회원 등록
    * /members/{id}
* 회원 수정
    * /members/{id}
* 회원 삭제
    * /members/{id}

근데 이렇게 하면 기능에 대한 구분을 어떻게 할 것인가?

### 리소스와 행위를 분리
가장 중요한 것은 리소스를 식별하는 것이다.

* URI는 리소스만 식별함. 행위 판별은 하지 않는다.
* 리소스와 행위를 분리한다.
  * 리소스: 회원
  * 행위: 조회 등록 수정 삭제
* 즉 리소스는 명사이며 행위는 동사임.
* 그렇다면 행위는 어떻게 구분하는가?
